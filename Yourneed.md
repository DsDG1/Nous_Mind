# Yourneed · 下一阶段 10 个发展方向

> 本文档整理 10 个**尚未实现但用户感知强、价值高**的功能方向，每项按"痛点 → 需求 → 验收 → 复杂度 → 依赖"五要素描述，可作为后续排期与立项的输入。已实现的同类功能（描述、AI 润色、通知 action 按钮）已落地于 v1.2.6，不再列入。

---

## 1. 重复提醒（Recurrence）

### 背景 / 现状
- 提醒是一次性的（`Reminder.reminderTime` 单一 `DateTime`），不支持"每天吃药""每周例会""每月缴费"等真实场景。
- 唯一相关能力是"稍后提醒"：手动把触发时间往后推 `SnoozeDuration`，不构成重复。

### 用户痛点
- 每周一/三/五 9 点的站会，每月 5 号还信用卡——目前每个周期都要手动新建一次；
- 重复创建导致列表里有大量"看上去过期但实际已经过去"的提醒，与 24h 自动删除产生混淆。

### 功能需求
- 新增 `repeat_rule: String?` 字段，使用 iCalendar `RRULE` 子集：
  - `FREQ=DAILY|WEEKLY|MONTHLY|YEARLY`
  - `INTERVAL=n`（每 n 天/周/月）
  - `BYDAY=MO,WE,FR`（仅 WEEKLY 有效）
  - `BYMONTHDAY=1,15`（仅 MONTHLY 有效）
  - `UNTIL=20261231T235959Z` 或 `COUNT=10`（两者必选其一；不设 → 无限）
- 编辑器新增"重复"入口，使用 Material 3 选择器：
  - 不重复 / 每天 / 每周 / 每月 / 每年 / 自定义
  - 自定义模式展开 RRULE 高级面板
- 通知触发后由后台逻辑（`flutter_local_notifications` 的 `onDidReceiveBackgroundNotificationResponse` 或 `WORKER`）按 `RRULE` 计算下次触发并重新 `zonedSchedule`。
- 数据库 schema 升至 v4：`ALTER TABLE reminders ADD COLUMN repeat_rule TEXT`。

### 验收标准
- 创建每日 9 点的提醒，下一天 9 点自动再次触发（无需 App 启动）；
- 每周一/三/周五 8:30 的提醒，三次都按时触发；
- 设置 `UNTIL=20260630` 后，7 月不再触发；
- 编辑器修改时间不影响既有 `repeat_rule`（除非用户主动改重复设置）；
- 备份导出/导入兼容旧数据（无 `repeat_rule` → 一次性）。

### 预估复杂度
**M**（约 1.5 周）—— RRULE 解析/计算用现成 `rrule` Dart 包；OS 调度+回调在 Android 需要 `WorkManager` 或前台的 `Timer`，iOS 同 `zonedSchedule`。

### 依赖
- `rrule: ^0.2.0`（社区实现 iCalendar RRULE）
- Android 14+ 的 `USE_EXACT_ALARM` 已声明，无需新权限
- iOS 无新权限

## 2. 分类 / 标签（Tags）

### 背景 / 现状
- 提醒和灵感都只有一个全局桶，列表按 `created_at DESC` 排，找不到"工作 vs 生活"分组。
- 任何分类/标签模型均不存在；首页没有筛选条。

### 用户痛点
- 100+ 条提醒后无法快速定位"工作"或"家庭"相关内容；
- 想批量查看本周"健康"类提醒（吃药、复诊、锻炼）需要逐条挑。

### 功能需求
- 标签模型：扁平 `tags: List<String>`，存到现有 `reminders`/`inspirations` 表的 JSON 列，无需新建 join 表。
- 编辑器底部加 `InputChip` 输入条：输入回车创建，标签可点击删除。
- 首页顶部加横向 chip 滚动条："全部" + 用户已有标签；选中后筛选。
- 多选筛选：长按 chip 进入多选模式，标题栏显示"已选 N 个标签"，支持"全部满足"或"任一满足"切换。
- 标签管理页（`设置 → 数据 → 标签管理`）：列出所有标签、每个标签下的提醒数；支持重命名/合并/删除（删除时把该标签从所有提醒中剥离）。

### 验收标准
- 给提醒打"#工作"标签，首页"工作" chip 只显示该类；
- 给同一提醒打多个标签，chip 多选时 OR/AND 行为符合切换；
- 重命名"#工作"为"#职场"，所有相关提醒的标签同步更新；
- 标签输入支持中文/英文/数字/下划线，最多 20 字符/标签，单提醒最多 10 标签。

### 预估复杂度
**M**（约 1 周）—— 主要是 UI 工作，模型改动小。

### 依赖
- 与"全文搜索"（#3）协同：搜索时可按 `tag:工作` 限定。

## 3. 全文搜索 + 状态筛选（Search & Filter）

### 背景 / 现状
- 提醒列表**无搜索**（灵感有但简陋）；
- 数据库 schema v1 曾创建 FTS5 virtual table，v1→v2 迁移时**主动 drop**；当前 `InspirationRepository.search` 退化为 `LIKE %query%`；
- OCR 文本没存库，AI 截图解析的结果不能被搜到。

### 用户痛点
- 几个月前的提醒想找回来只能逐条翻；
- 截图里的"项目名/电话号码"想搜，却根本没索引。

### 功能需求
- 数据库 schema v4 升 v5：恢复 FTS5 virtual table
  - `CREATE VIRTUAL TABLE reminders_fts USING fts5(id UNINDEXED, title, description, ocr_text, content=reminders, content_rowid=rowid)`
  - 通过 triggers 同步增删改
- OCR 缓存：AI 助手分析截图时把 `TextRecognitionScript` 结果落到 `reminders.ocr_text`（新增列），下次搜索直接命中。
- 首页搜索栏（`SearchDelegate` 模式，参考灵感页）：
  - 多字段：title / description / ocr_text
  - 语法：`tag:工作` 限定标签，`before:20260630` 限定日期范围
  - 高亮匹配片段
- 状态筛选 chip：全部 / 待触发 / 已过期 / 24h 内 / 有图 / 有描述。
- 排序选项：创建时间（默认）/ 触发时间 / 标题。

### 验收标准
- 搜"买菜"命中所有含此关键字的提醒（含描述中）；
- 截图里的"2026 Q3 OKR"被识别后，搜"OKR"命中该提醒；
- `tag:健康 before:20260701` 只返回健康类且 7 月前的；
- 排序"触发时间"在"今天/明天/本周/更晚"分组的视图中正确。

### 预估复杂度
**L**（约 2 周）—— FTS5 迁移 + OCR 缓存列 + 复杂搜索语法解析。

### 依赖
- 与 #2 标签强协同
- 与 #1 重复提醒的"下一次触发时间"排序需要 RRULE 算子

## 4. 子任务 / 清单（Subtasks / Checklist）

### 背景 / 现状
- 一个提醒只有 `title` + `description`，无法承载"出门带：钥匙、手机、钱包、充电器"这种清单场景。
- 灵感不支持任何结构化（只有纯文本）。

### 用户痛点
- 出门/出差前的"检查清单"目前是长字符串，难以勾选已完成的项；
- 通知弹出时只能看标题，看不到完成进度。

### 功能需求
- 新增 `subtasks` 表：`id`/`reminder_id`/`text`/`is_done`/`order`/`created_at`
- 编辑器在描述下方加"清单"模块：
  - 顶部按钮"+ 添加条目"
  - 每行：勾选框 + 文本（点击编辑）+ 拖拽手柄 + 删除
  - 已勾选的项自动划线 + 移到底部
- 通知 body 用子任务进度渲染：
  - "出门带：✅ 钥匙 ✅ 手机 ⬜ 钱包 ⬜ 充电器（2/4）"
- 列表项右侧加完成度环（如 3/5 进度圆环）。

### 验收标准
- 创建含 5 条子任务的提醒，全部勾选后通知 body 显示"5/5"；
- 拖拽改顺序，顺序持久化；
- 删除某条子任务，列表行同步消失；
- 无子任务时，body 仍回退到描述（如有）或默认后缀。

### 预估复杂度
**M**（约 1 周）—— 一张新表 + 编辑器拖拽组件。

### 依赖
- 与 #3 全文搜索协同：子任务文本应可被搜到

## 5. 软删除 / 回收站（Trash）

### 背景 / 现状
- 删除提醒即真删，连同图片文件。
- 24h 自动删除无任何恢复机会。
- `clearAll` 同样不可逆。

### 用户痛点
- "啊我手滑了"——目前只能从备份恢复，门槛高；
- 团队/家庭共享设备时小孩误删，恢复困难。

### 功能需求
- `Reminder` 增加 `is_deleted: bool` + `deleted_at: DateTime?`（替代硬删除路径，保留 30 天）
- UI 路径：
  - 单条删除 → 弹出"已移入回收站" SnackBar + 5 秒"撤销"按钮
  - `clearAll` → 先全部进回收站，再弹一次"清空回收站"二次确认
  - 列表加"回收站"入口，显示已删项 + 剩余天数
- 回收站页：
  - 列表显示"将在 X 天后清除"（基于 `deleted_at + 30d`）
  - 顶部按钮"全部恢复" / "全部永久删除"
- 永久删除路径仍存在（从回收站页执行）。
- 数据库 v4：增加 `is_deleted` + `deleted_at` 列 + 索引。

### 验收标准
- 删除提醒后 30 天内可从回收站恢复，恢复后通知调度同步重排；
- 30 天后自动清理（与现有的 `_purgeExpiredReminders` 类似机制）；
- 永久删除会清理图片文件（同现有 `delete` 逻辑）；
- 备份导出包含"是否包含回收站"开关（默认否）。

### 预估复杂度
**M**（约 1 周）—— 主要是 UI 路径分流 + 软删除索引。

### 依赖
- 与备份兼容：旧备份无 `is_deleted` → 视为未删除

## 6. 主页 / 锁屏 Widget

### 背景 / 现状
- Android 仅有"快速添加"QS Tile（点开 App 创建），没有任何"看一眼下一个提醒"的入口；
- iOS 无对应功能；锁屏也没有任何信息呈现。

### 用户痛点
- 想看"我下一个该做什么"必须解锁 → 找 App → 启动，路径太长；
- "今天还有几个提醒"无法一眼判断。

### 功能需求

**Android AppWidget**
- 新建 `ReminderWidgetProvider extends AppWidgetProvider`
- 小尺寸（2x2）：显示下一个未触发提醒的标题 + 触发时间
- 中尺寸（4x2）：显示今天所有提醒列表（最多 5 条）
- 大尺寸（4x4）：本周日历缩略（与 #7 日历视图共用组件）
- 点击：跳到 App 对应提醒详情

**iOS WidgetKit**
- 三种 size 同上
- 通过 `WidgetCenter.shared.reloadAllTimelines()` 在 `add/update/delete` 后触发刷新
- iOS 16+ 支持 `LiveActivity` 在锁屏显示"距离触发还有 X 分钟"（仅未触发时）

### 验收标准
- 在主屏添加 widget，1 秒内显示下一个提醒；
- 新建提醒后 widget 自动刷新（无需重启 App 或主屏）；
- 中尺寸 widget 滚动可见更多条；
- 删除全部提醒后 widget 显示"暂无提醒"占位。

### 预估复杂度
**L**（约 2-3 周）—— 涉及原生开发（Kotlin AppWidget + Swift WidgetKit），且需在 Flutter 侧暴露数据接口。

### 依赖
- 与 #7 日历视图共用"今日/本周"组件
- 与 #1 重复提醒：widget 必须能正确显示"下一次触发时间"

## 7. 日历 / 议程视图（Calendar / Agenda）

### 背景 / 现状
- 提醒列表恒为 `created_at DESC` 排序，"明天到期的提醒"可能被上周创建的其他提醒压在下面。
- 无月历/周视图/议程视图。

### 用户痛点
- 看不出"今天该做什么"；
- 想知道"下周有什么安排"必须一条条挑。

### 功能需求
- 主页加 segmented control：列表 / 日历
- 日历模式（基于 `table_calendar` 包）：
  - 月视图：日期格子右下角红点表示"有提醒"（角标数字显示数量）
  - 周视图：横滑切换周，垂直时间轴
  - 日视图：当日 24h 时间轴，drag-to-create
- 议程模式：自动按"今天/明天/本周/本月/更晚"分组
- 跨视图拖拽：长按提醒拖到日历某天 = 修改触发日期

### 验收标准
- 月视图正确显示 6 月 14 日的"买菜"提醒（红点 + 1）；
- 切到日视图，6 月 14 日显示 18:00 提醒一行；
- 在日历模式点空白日期 → 跳到新增提醒页，预填日期；
- 议程模式正确把 5 天前的过期提醒分到"已过期"组（不再依赖"自动删除 24h"）。

### 预估复杂度
**L**（约 2 周）—— `table_calendar` 集成 + 自定义 painter 绘制提醒点。

### 依赖
- 与 #6 Widget 共用月视图组件
- 与 #1 重复提醒：日历格子需要展开"重复事件"到具体日期

## 8. i18n 国际化（Internationalization）

### 背景 / 现状
- 全量 UI 字符串硬编码中文（grep 显示 `lib/` 内千余处中文字面量）；
- 无 `flutter_localizations`、无 `.arb`、无 `intl` 包；
- 任何想用英文/日文/其他语言的用户被迫面对中文界面。

### 用户痛点
- 海外华人/留学场景下中文界面不友好；
- 完全没有"未来出海"的物质基础。

### 功能需求
- 添加 `flutter_localizations` + `intl`，定义 `MaterialApp.localizationsDelegates`
- 新建 `lib/l10n/` 目录，提取 `app_en.arb`（基准英文）+ `app_zh.arb`（现有中文）
- 添加 `Settings → 外观 → 语言`：跟随系统 / 中文 / English
- 关键字符串清单（按出现频次）：
  1. 页面标题（提醒事项、AI 助手、灵感、设置、5 个子页）
  2. 通用按钮（保存、取消、删除、确定、稍后提醒、完成）
  3. 错误消息（API 密钥无效、网络异常、…）
  4. 通知 body（"提醒时间到啦"）
- 日期/时间格式接入 `intl.DateFormat`（不再用 `_formatDateTime` 硬编码 `YYYY-MM-DD HH:mm`）
- 备份 JSON 携带 `locale` 字段，导入时尊重目标设备设置

### 验收标准
- 切换到 English 后所有页面变成英文；
- 通知 body 在系统语言为英文时显示 "Time's up"；
- 日期在英文 locale 下显示 "Jun 14, 2026 6:30 PM"；
- 任何遗漏的中文串会被审计脚本（CI 中跑 `grep -rE '[\xe4-\xe9][\x80-\xbf]{2}' lib/l10n/app_en.arb`）抓到。

### 预估复杂度
**XL**（约 4-6 周）—— 字符串数量大，需要逐文件替换 + 维护双 .arb。

### 依赖
- 与 #3 搜索：搜索结果的日期格式跟随 locale
- 与 #7 日历：月历标题依赖 locale

## 9. 图片纳入备份（Images in Backup）

### 背景 / 现状
- `BackupService` 导出 JSON 引用 `imagePath` 字符串，但**图片文件本身不打包**。
- 用户换机/重装 → 导入后所有图片"已删除"占位（`Icons.broken_image`）。

### 用户痛点
- "我以为备份了，结果图片全没了"——最常见的迁移事故；
- 备份文件几百 KB（小），但用起来跟"半残"一样。

### 功能需求
- 导出：把 `<appDocs>/inspiration_images/` 全部文件 + JSON 一起打成 zip
  - 文件名 `nous_backup_<ts>.zip`
  - 内部结构：`backup.json` + `images/<id>.<ext>`
- 导入：解压 zip，把图片复制回 `<appDocs>/inspiration_images/`
  - 缺失图片 → 提醒里 `imagePath` 置 null
- 保留旧版"纯 JSON" 备份格式的导入兼容（检测 zip 头 magic number `PK`）
- 备份页 UI 加进度条（zip 压缩 + 复制都是 IO 密集）
- 取消 share sheet 时清理 tmp 文件（目前依赖 OS）

### 验收标准
- 导出含 50 张图的备份，zip 大小比 JSON+图片总和略小；
- 卸载 App → 重装 → 导入 zip，图片完整恢复；
- 旧版 JSON 备份仍可导入（图片缺失会标记在统计里"X 条提醒无图"）；
- zip 加密（可选）：设置开关"AES 加密 + 密码"。

### 预估复杂度
**M**（约 1 周）—— `archive` 包已成熟，逻辑直白。

### 依赖
- 与 #10 自动备份强协同
- 可选加密：引入 `cryptography`/`pointycastle` 包

## 10. 手动编辑器自然语言时间输入（NL Time Parsing）

### 背景 / 现状
- 手动编辑器只能用 `showDatePicker` + `showTimePicker` 拨几圈；
- "明天下午 3 点"必须拨 4 次（明天→日期→下午→3:00）。
- AI 助手能识别 NL 时间，但需要联网 + 消耗 token + 走完整流程（截图/粘贴 → 解析 → 选草稿），大多数用户不愿绕一圈。

### 用户痛点
- 设个简单时间要拨好几圈；
- 复杂表达"下周三下午 4 点半"在 picker 里极繁琐。

### 功能需求
- 编辑器时间卡片上叠一个"⌨ 直接输入"入口（TextField + 回车解析）
- 本地正则 + Dart `intl` 解析常见模式（**离线、零延迟、零 token**）：
  - `今天/明天/后天 + X点[Y分]`
  - `周X + X点[Y分]`（如"周三下午3点"）
  - `X月X日 + X点[Y分]`
  - `X 分钟/小时/天后`
  - `上午/下午/早上/晚上` 修饰
- 解析成功 → 时间卡片直接更新；解析失败 → 显示"未能识别"提示并保留 picker 入口
- 高级模式（可选）：把"无法本地解析"的输入送 AI 助手兜底，标记为"网络辅助"，成功后用同样的 UI 回填
- 解析库自研（轻量正则集）即可，不引入额外包

### 验收标准
- 输入"明天下午3点" → 时间更新为明天 15:00；
- 输入"下周三下午4点30" → 正确跳到下周三 16:30（注意跨周逻辑）；
- 输入"3小时后" → 当前时间 + 3h；
- 输入"2026/12/31 23:59" → 直接当 ISO 解析；
- 输入"asdfasdf" → 红色提示 + 时间卡片不变。

### 预估复杂度
**M**（约 1 周）—— 解析器 ~200 行 Dart 正则，覆盖 80% 常见表达。

### 依赖
- 与 #1 重复提醒：第一次触发时间若用 NL 表达，UI 仍显示 picker 时间
